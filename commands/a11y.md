## Accessibility Audit

Audit UI code for accessibility issues against WCAG 2.2 AA standards. Review markup, styles, and behavior for barriers that exclude users of assistive technologies.

### Scope

Target: $ARGUMENTS

If no scope is provided, scan staged changes or the most recently modified UI files.

### What to Check

Audit every UI component, template, and style change for the following:

1. **Semantic Structure**
   - Landmarks: `<main>`, `<nav>`, `<header>`, `<footer>`, `<aside>` used correctly
   - Heading hierarchy: no skipped levels (`h1` → `h3` without `h2`), only one `h1` per page
   - Lists: `<ul>`, `<ol>`, `<dl>` used for grouped items instead of styled divs
   - Tables: `<th>` with `scope` attribute, `<caption>` present, no layout tables
   - `<div>` and `<span>` only when no semantic element exists — prefer `<button>`, `<a>`, `<fieldset>`

2. **Keyboard Access**
   - All interactive elements reachable via Tab (no `tabindex="-1"` on actionable elements)
   - Custom controls have correct `tabindex="0"` and keyboard handlers (Enter, Space, Arrow keys)
   - Focus order matches visual order — no `tabindex` values > 0 creating custom tab sequences
   - No focus traps without escape mechanism (modals, dialogs)
   - Focus is managed after dynamic content changes (route transitions, form submissions)
   - `:focus-visible` styles exist and are visible — `outline: none` is paired with a visible alternative
   - Skip-to-content link at top of page

3. **ARIA & Roles**
   - ARIA only when HTML isn't enough — no `role="button"` on `<button>`, no `role="link"` on `<a>`
   - Custom widgets have correct roles: `tab`, `tabpanel`, `tablist`, `menu`, `menuitem`, `dialog`, `alertdialog`
   - `aria-label` or `aria-labelledby` on all interactive elements without visible text (icon buttons, inputs)
   - `aria-expanded` on expandable elements, `aria-selected` on selected items
   - `aria-live` regions for dynamic content: `polite` for updates, `assertive` for alerts/errors
   - `aria-hidden="true"` on decorative elements only — never on focusable or interactive elements
   - `aria-describedby` for form fields with helper text or error messages

4. **Forms & Inputs**
   - Every input has an associated `<label>` (explicit `for`/`id` or implicit wrapping)
   - Placeholder text is not used as the only label
   - Required fields marked with `required` attribute AND visible indicator — not color alone
   - Error messages are programmatically associated with inputs via `aria-describedby`
   - Error states are announced to screen readers (`aria-live` or focus management)
   - `<fieldset>` + `<legend>` for related groups (radios, checkboxes)
   - Autocomplete attributes present (`autocomplete="email"`, `autocomplete="tel"`, etc.)
   - Form validation errors are surfaced accessibly, not only with red borders

5. **Color & Contrast**
   - Text meets minimum contrast ratio: 4.5:1 for normal text, 3:1 for large text (18px+ or 14px bold+)
   - Non-text UI elements (icons, borders, focus indicators) meet 3:1 contrast against adjacent colors
   - Information is never conveyed by color alone — there's always a text/icon/shape indicator too
   - Dark mode and light mode both have sufficient contrast
   - Link text is distinguishable from surrounding text by more than just color (underline, icon, weight)

6. **Images & Media**
   - All `<img>` elements have `alt` text: descriptive for content images, empty `alt=""` for decorative images
   - Complex images (charts, diagrams) have extended descriptions nearby or via `aria-describedby`
   - SVGs have `<title>` and `<desc>`, `role="img"`, `aria-labelledby`
   - Video has captions, audio has transcripts, media has accessible controls
   - No auto-playing video/audio lasting more than 3 seconds without a pause/stop control

7. **Motion & Timing**
   - Animations respect `prefers-reduced-motion` media query — disable or reduce non-essential motion
   - Auto-advancing carousels/sliders have pause, stop, and manual navigation controls
   - Session timeouts provide warning and extension option
   - No content that flashes more than 3 times per second (seizure risk)
   - Scroll-driven animations are gated behind `prefers-reduced-motion: no-preference`

8. **Touch & Pointer**
   - Touch targets are at least 44×44px (WCAG 2.5.5) — or 24×24px if equivalent target exists
   - Adequate spacing between interactive elements on mobile
   - No reliance on hover-only interactions — touch-first alternatives
   - Gestures have fallbacks (pinch-to-zoom alternatives, swipe alternatives)

9. **Screen Reader UX**
   - Page `<title>` is unique and descriptive
   - `lang` attribute set on `<html>` element, `lang` on inline language changes
   - Visually hidden text for screen reader context (".sr-only", ".visually-hidden" patterns)
   - Toast notifications use `role="status"` or `role="alert"` with `aria-live`
   - Modal dialogs trap focus and Close button is keyboard accessible

10. **Dynamic Content & SPA Concerns**
    - Route changes trigger focus move to new content (heading or main container)
    - Loading states are announced to screen readers
    - Infinite scroll has a "load more" button alternative
    - Client-rendered content doesn't create a gap between DOM order and visual order
    - Skeleton loading states have `aria-busy` or `aria-hidden` to avoid confusing screen readers

### Output Format

Group by severity:

- **Blocker** — User literally cannot complete the task (missing label on input, button unreachable by keyboard, required information only conveyed by color)
- **Major** — Significant barrier that forces workarounds (missing heading hierarchy, poor contrast, focus not managed after navigation)
- **Minor** — Degraded experience but task is possible (missing autocomplete, non-ideal ARIA usage, touch target slightly small)
- **Advisory** — Best practice improvements (skip link location, prefers-reduced-motion polish, semantic element opportunities)

For each finding:
- **Severity**
- **File:Line** — exact location of the markup/style
- **WCAG SC** — relevant Success Criterion (e.g., 1.1.1 Non-text Content, 1.3.1 Info and Relationships, 2.1.1 Keyboard)
- **Issue** — what's wrong and who it affects
- **Fix** — concrete code change, with before/after snippet if helpful

### Important

- Don't flag issues you can't verify from code alone (actual contrast ratios require rendered output — but flag hardcoded low-contrast color pairs)
- Framework specifics: React uses `htmlFor` not `for`; Vue uses `v-bind` patterns; note these in your analysis
- CSS-in-JS and utility-first frameworks (Tailwind) are fine — check the resulting HTML semantics regardless of how it's authored
- If no issues found, confirm and note that manual testing with actual screen readers is still recommended
